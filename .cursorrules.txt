# [PROJECT OVERVIEW]
Project Name: "Aether"
Description: A completely decentralized, serverless Desktop P2P social network and messenger. The application runs as a standalone local node.
Terminology:
- "User ID" = The user's Public Key.
- "Post" = A JSON object signed by the user's Private Key, distributed via libp2p pubsub.
- "Follow/Subscribe" = Adding another user's Public Key to the local database to listen for their pubsub messages.

- Backend (Main Process): Node.js.

- Local Database: `sqlite3` or `level` (running purely in the Electron Main process). *Note: Be mindful of native Node modules in Electron; ensure proper rebuild processes (e.g., `electron-rebuild` or `electron-builder` native deps compilation) are suggested and handled if needed.*
- Testing: Vitest.

# [CORE ARCHITECTURE & CONSTRAINTS]
1. NO CENTRAL SERVERS: Do not use Firebase, Supabase, AWS, or any centralized databases/REST APIs.
2. P2P ONLY: All communication must happen directly between peers or via decentralized relays.
3. LOCAL FIRST: All user data, keys, and cached feeds must be stored locally on the user's hard drive. E2EE for Direct Messages is mandatory.
4. STRICT IPC COMMUNICATION: The React frontend MUST NOT handle P2P networking or cryptography directly. The frontend only communicates with the Electron Main process via `contextBridge` (IPC). 
5. SECURE KEY MANAGEMENT: Private keys must NEVER leave the user's machine. Generate them locally, store them securely in the OS keychain or an encrypted local DB, and use them only for local signing.
